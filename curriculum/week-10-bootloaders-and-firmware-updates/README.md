# Week 10 — Bootloaders and Firmware Updates on the RP2040

> *A bootloader is the program that decides which program runs next. It is small, it is paranoid, and it is the one piece of firmware on the chip that cannot have a bug — because if it does, the device is a brick and the only recovery is the soldering iron. Every device in your house with a USB port and an update mechanism is running one. Most of them are written in C, occupy fewer than 32 KB of flash, and follow a recipe that has not meaningfully changed since the 1980s: validate, copy, jump, pray. The interesting part is that on the RP2040, the first stage of this recipe is in mask ROM and you cannot change it; the second stage is the 256 bytes at the start of your flash image and you write it every time you build; and the third stage — the one that does OTA updates and signature verification and dual-bank swaps — is the program you are about to write.*

Welcome to Week 10 of C7. Last week your Pico enumerated as three USB devices at once — a CDC virtual serial port, an HID keyboard, and a 64 KB MSC RAM disk — through a 161-byte composite configuration descriptor that you computed `wTotalLength` for at compile time. You learned that the silicon below TinyUSB handles NRZI and bit-stuffing and CRC16; that the descriptor work above it is byte-perfect bookkeeping; and that one wrong `bLength` collapses enumeration. This week you remove TinyUSB from the picture for a moment and ask a different question: how does the firmware get onto the chip *in the first place*, and how do you replace it once the device is in the field and nobody can plug in a UF2 drag-and-drop?

The boot sequence on the RP2040 is documented in RP2040 datasheet §2.8, pp. 130–141 — eleven pages, three figures, and a state diagram. When the RP2040 leaves reset, core 0 executes from address `0x00000000` which is mapped to the on-chip Boot ROM. The Boot ROM is a 16 KB mask-ROM image baked into the silicon at fab time (datasheet §2.8.1, p. 130); you cannot patch it, you cannot read its source (Raspberry Pi did not publish it), but its behavior is fully specified. The Boot ROM does five things in order: it sets up the clocks to the default 6.5 MHz ring-oscillator pace, it checks GPIO 23 (the BOOTSEL line is shared with the QSPI chip-select on the Pico board, hence the dual function), it tries to load a 256-byte *second-stage bootloader* from QSPI flash at offset 0, it CRC32-checks that 256 bytes, and if the CRC passes it jumps to the user image at flash offset 256. If the CRC fails *or* if BOOTSEL is held during reset, the Boot ROM falls back to its USB MSC mode: it enumerates as a 128 MB drive named `RPI-RP2`, accepts UF2 files dropped on it, and writes their payloads to flash.

This UF2 mechanism is what you have been using since Week 1 of C7. It is a beautifully simple recovery scheme — drop a file on a drive, the chip programs itself, reboots into the new firmware — and it is the *baseline* firmware-update story for any RP2040 device that ships with the BOOTSEL button exposed. For a Pico on a breadboard that is fine. For a sealed industrial sensor in a junction box on a roof, BOOTSEL is not exposed; the only update path is over whatever interface the device already has — USB, WiFi, BLE, LoRa, RS-485. This week we build the second mechanism: a *bootloader* that lives in your flash (not in the mask ROM, you cannot touch that), validates a downloaded image, copies it from staging to active, and jumps to it. Plus the host-side machinery — UF2 format, USB DFU class spec, a signed-image format with an Ed25519 signature — that makes the receive side trustworthy.

The bootloader you will write fits in 32 KB and lives at flash offset 0x10000000 + 0x100 (the first 256 bytes are still the RP2040 second-stage bootloader, which the Boot ROM CRC32-checks). Above the bootloader's 32 KB are two 768 KB *banks*: the active bank at offset 0x8100 holds the firmware currently running; the staging bank at offset 0xC4100 holds the firmware that was just downloaded and is awaiting validation. At reset, the bootloader: (1) reads the *metadata* page at offset 0xFC000 to see if a "swap requested" flag is set; (2) if yes, it validates the staging image's Ed25519 signature against a public key stored in the bootloader itself; (3) if the signature is good, it copies staging into active, clears the swap flag, and reboots; (4) if no swap is requested, it validates the active image's signature; (5) if active validates, it jumps to it; (6) if active fails validation, it tries staging as a last-resort fallback; (7) if both fail, it falls through to BOOTSEL/USB-MSC recovery mode by jumping back into the Boot ROM's USB code at the documented entry point in datasheet §2.8.5, p. 138.

The mini-project is a complete bootloader plus a host-side `cc-flash` Python tool. The bootloader is the C program described above, with the signature verifier using a small Ed25519 implementation (Bernstein/Lange/Schwabe reference C code, public domain, ~1500 LOC) and the flash-write code calling the Boot ROM's `flash_range_program` function (the Boot ROM exposes a small API at known addresses, documented in datasheet §2.8.3.1, p. 132). The host tool reads a `.elf` from the Pico SDK build, extracts the firmware bytes, computes the Ed25519 signature with a private key stored in `~/.cc-flash-keys/`, packages the signed image into a custom `.ccf` (Code Crunch Firmware) format, and transmits it over the CDC serial port using a tiny chunked protocol. On the device side, the user firmware (the *application*, not the bootloader) implements a `cc-flash receive` command: it reads the `.ccf` chunks from CDC, writes them to the staging bank, sets the swap flag, and reboots. The bootloader takes over from there. End to end, you can `cc-flash my_firmware.elf` from your laptop and the Pico is running the new firmware ~12 seconds later — no BOOTSEL button required.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** the RP2040's three-stage boot sequence. Stage 0 is the Boot ROM at `0x00000000` (RP2040 datasheet §2.8.1, pp. 130–131): 16 KB of mask ROM that handles cold reset, watchdog reset, and BOOTSEL detection. Stage 1 is the 256-byte second-stage bootloader at flash offset 0 (datasheet §2.8.1.2, p. 132): this is the only piece of code that knows how to configure the QSPI flash for execute-in-place (XIP) mode, and the Pico SDK ships several variants (`boot2_w25q080.S` for the Pico's W25Q16JV-IQ flash chip, plus boot2 variants for Adesto, Generic 03h, etc., in the SDK at `src/rp2_common/boot_stage2/`). The Boot ROM CRC32-checks the first 252 bytes against the trailing 4-byte CRC; if it matches, the ROM `bx`-jumps to flash offset 0. Stage 2 is your bootloader, beginning at flash offset 256 (`0x10000100`), where the Stage 1 bootloader transfers control after setting up XIP.
- **Read** the UF2 file format. UF2 (USB Flashing Format) is Microsoft's drag-and-drop firmware format used by the RPI-RP2 MSC drive, the Adafruit nRF52 line, the BBC micro:bit, and many CircuitPython boards. Spec at <https://github.com/microsoft/uf2> (BSD-2-Clause, ~600 lines of README and ~200 lines of header). A UF2 file is a sequence of 512-byte blocks; each block has a 32-byte header (`magicStart0=0x0A324655u, magicStart1=0x9E5D5157u, flags, targetAddr, payloadSize, blockNo, numBlocks, fileSize_or_familyID`), 476 bytes of payload (the first `payloadSize` of which are valid; typically 256 for RP2040), and a 4-byte trailer (`magicEnd=0x0AB16F30u`). The RP2040's family ID is `0xE48BFF56u`, registered in `microsoft/uf2/utils/uf2families.json`. Tools that pack UF2 (the Pico SDK's `elf2uf2`, the `uf2tool` npm package, etc.) walk the ELF's loadable sections and emit one UF2 block per 256-byte slice. You will write a 120-line `uf2.c` parser this week as Exercise 1.
- **Explain** the USB DFU class. Device Firmware Upgrade is USB-IF class code 0xFE/01/0x{00,01,02} for run-time/DFU-mode descriptors; spec is "USB Device Firmware Upgrade Specification, Revision 1.1" (free at <https://www.usb.org/sites/default/files/DFU_1.1.pdf>, 47 pages, dated 2004-08-05). DFU has two modes: in *run-time mode* the device exposes its normal application interfaces plus one DFU interface; the host sends `DFU_DETACH` over the DFU interface and the device re-enumerates in *DFU mode* with only the DFU interface present; the host then runs the seven DFU requests (`DFU_DNLOAD`, `DFU_UPLOAD`, `DFU_GETSTATUS`, `DFU_CLRSTATUS`, `DFU_GETSTATE`, `DFU_ABORT`, `DFU_DETACH`) on endpoint zero to transfer the firmware. The state machine has ten states (`appIDLE`, `appDETACH`, `dfuIDLE`, `dfuDNLOAD_SYNC`, `dfuDNBUSY`, `dfuDNLOAD_IDLE`, `dfuMANIFEST_SYNC`, `dfuMANIFEST`, `dfuMANIFEST_WAIT_RESET`, `dfuERROR`) documented in DFU 1.1 §6, pp. 16–25. We will implement a *subset* DFU-mode device in Exercise 3 — read-only download, no upload, no manifest tolerance — to keep the code under 400 lines.
- **Design** a dual-bank update scheme. The classical approach (used by MCUboot, FreeBSD's `loader.efi`, and most automotive ECUs) is *A/B partitioning*: flash is split into a "slot A" and a "slot B" of identical size; one is currently running, one is being written to; on validation, the bootloader switches which slot it boots from next time. Variants: *swap-and-run* (copy B to A, run A, copy A back to B on first successful boot — survives a power loss mid-update because there are always two intact copies), *boot-from-either* (the bootloader picks the slot with the higher version number that also signature-verifies — needs metadata pages with version numbers but avoids the copy), and *staging-overlay* (one full-size active bank plus a smaller staging bank that the bootloader copies to active in one shot — the simplest, what we build this week). Cite MCUboot's design doc at <https://docs.mcuboot.com/design.html> for the canonical treatment.
- **Implement** a flash layout. The RP2040 has 2 MB of QSPI flash (on the standard Pico; some boards have 4 MB or 16 MB). Our layout for this week's project: `[boot2: 0–0xFF (256 bytes)] [bootloader: 0x100–0x80FF (~32 KB)] [active app: 0x8100–0xC40FF (~752 KB)] [staging app: 0xC4100–0x180FFF (~752 KB)] [metadata: 0xFC000–0xFFFFF (16 KB, two 4 KB pages for swap/version state)]`. The linker script (the Pico SDK's `memmap_default.ld`, which we patch into `memmap_app.ld` and `memmap_bootloader.ld`) places `.text` at the right offset per build. The flash erase sector is 4 KB and the program page is 256 bytes — both enforced by the W25Q16JV-IQ chip (Winbond datasheet at <https://www.winbond.com/resource-files/W25Q16JV_DTR_RevD%2008292016.pdf>, §7.2.18 Sector Erase pp. 17, §7.2.10 Page Program pp. 32). The bootloader's flash operations must respect these alignments.
- **Use** the Boot ROM's flash API. The Boot ROM exposes a tiny table of function pointers at flash offset `0x00000000 + 0x14` (datasheet §2.8.3.1, pp. 132–134). The functions we need: `_flash_range_erase(addr, count, block_size, cmd)` erases `count` bytes starting at `addr` using `cmd` as the QSPI sector-erase opcode; `_flash_range_program(addr, data, count)` programs `count` bytes from `data` into flash at `addr`; `_connect_internal_flash()` and `_flash_exit_xip()` to leave XIP mode before erase/program (required — you cannot read flash while writing it); `_flash_flush_cache()` to evict the XIP cache after a write; `_flash_enter_cmd_xip()` to re-enter XIP. The Pico SDK wraps these as `flash_range_erase` and `flash_range_program` in `hardware_flash`. You will call the SDK wrappers from the application side and the Boot ROM functions directly from the bootloader side (because the bootloader cannot link the SDK's flash helpers without bloating its 32 KB).
- **Implement** an Ed25519 signature verifier. Ed25519 is the Edwards-curve Digital Signature Algorithm at the 128-bit security level (RFC 8032, <https://www.rfc-editor.org/rfc/rfc8032.html>, §5.1). The reference C implementation is by Bernstein, Lange, and Schwabe in their `ed25519` directory of the SUPERCOP benchmark suite (<https://bench.cr.yp.to/supercop.html>, public domain). For embedded use, there are smaller versions: the `monocypher` library (<https://monocypher.org/>, 2-clause BSD, ~2000 LOC for the whole library, ~600 LOC for Ed25519 sign+verify), the `ed25519-donna` library (<https://github.com/floodyberry/ed25519-donna>, public domain, ~1500 LOC for verify-only), and the WolfSSL `wc_ed25519.c` (GPL or commercial). We use a verify-only fork of ed25519-donna in this week's mini-project; it compiles to ~12 KB of `.text` at `-Os` on Cortex-M0+ which fits comfortably in the bootloader's 32 KB budget. The bootloader stores the project's public key (32 bytes) as a `const uint8_t` array.
- **Design** an OTA update protocol. "Over-the-air" is the umbrella term for any non-physical-button update path. For the RP2040 in this week's project, "OTA" is over the USB CDC port (the Pico is plugged into a host but the host runs a *user* program, not a programmer like `picotool`). The protocol: host sends `BEGIN <size> <sha256>`; device acknowledges with `OK`; host sends `CHUNK <offset> <len> <data>` repeatedly; device writes each chunk to staging flash and acknowledges with `OK <offset>`; host sends `END`; device verifies the SHA256 against the announced one, sets the swap flag in metadata, sends `REBOOT`, and calls the SDK's `watchdog_reboot` to trigger a reset. The bootloader takes over from there. The protocol is line-oriented ASCII with hex-encoded binary chunks; it is ugly and slow (~5 KB/s) but trivial to debug with `screen /dev/cu.usbmodem...`. Cite the MCUboot serial-recovery protocol at <https://docs.mcuboot.com/serial_recovery.html> for inspiration; we simplified it.
- **Reason** about anti-bricking and rollback. The two failure modes that destroy a device: (1) the new firmware boots but the app inside it is broken — no UART output, no USB, no GPIO toggling, the user has no way to invoke the update path again; (2) the new firmware does not boot at all — the bootloader jumps to the `Reset_Handler` entry point and the chip faults immediately. Mitigations: a "first-boot confirmation" flag the new firmware must set within N seconds or the bootloader reverts on the next reset; a watchdog the new firmware must kick; a separate "fallback" partition that the user can force via a physical input (the BOOTSEL button if exposed, or a reset-count-based fallback — three rapid resets triggers fallback mode). The MCUboot project documents all three under "Image swap with rollback" (<https://docs.mcuboot.com/design.html#image-swap-with-rollback>) and "Direct-XIP" (no swap, just point-of-execution). We implement the watchdog-based confirmation in the mini-project.
- **Inspect** the RP2040's `picotool` and the Boot ROM USB MSC mode. `picotool info -a` (Raspberry Pi tool, <https://github.com/raspberrypi/picotool>, BSD-3-Clause) prints the partition table, the program info embedded in the binary (via `bi_decl(...)` macros), the address of the binary, and the SDK version. The `picotool reboot -u -f` command soft-resets the chip into USB MSC mode without touching BOOTSEL. We will use `picotool` as our debugging telescope all week — it reads everything from the chip non-invasively (the chip exposes a vendor-specific reset interface via USB when the SDK is built with `pico_set_binary_type(... pico_default)`).
- **Read** RP2040 datasheet §2.8 in full. Eleven pages, three figures, one state diagram. You will read this twice this week. The first read is "what does the Boot ROM do at reset?" — the answer is on pp. 130–131. The second read is "what does the Boot ROM provide my bootloader at runtime?" — the answer is the function table at pp. 132–134 and the entry-point list at p. 138. Treat this as the Week 9 USB 2.0 Chapter 9 of this week.
- **Build** a complete bootloader plus signed-image plus host-side `cc-flash` tool. The bootloader is 12 source files (header + 6 C files + 5 support files), ~1800 LOC, fits in 28 KB at `-Os` with the verify-only Ed25519 included. The host tool is a 200-line Python 3 script using `pyserial` and the `cryptography` library's `Ed25519PrivateKey`. End-to-end: `make` builds bootloader, application, and the application is signed by `cc-sign`; `cc-flash app.signed.bin` uploads it over CDC; the device reboots and runs the new firmware in ~12 seconds. Budget: bootloader < 32 KB, application + bootloader together < 800 KB so we leave > 1.2 MB for two banks plus metadata.

## Topics covered

- **The boot sequence in detail.** Cold reset → Boot ROM at `0x00000000` → clock setup → BOOTSEL check (GPIO 23 sampled while QSPI CS is asserted) → if held, USB MSC mode; if released, CRC32 of flash[0..251] against flash[252..255] → if mismatch, USB MSC mode; if match, `bx` to flash[0] → Stage 1 bootloader sets up XIP, jumps to flash[256] → user code (or our bootloader). Cite RP2040 datasheet §2.8.1, pp. 130–132 and Figure 11 on p. 131. The transition from Boot ROM to Stage 1 happens with the CPU clock still at 6.5 MHz (the ROSC default); Stage 1's `boot2_w25q080.S` configures XIP and *some* boards set a faster clock here, but the Pico SDK's variant leaves clock setup to the user program.
- **The 256-byte second-stage bootloader.** Stage 1's job is to configure the QSPI flash for execute-in-place. The W25Q16JV-IQ supports `0xEB` Quad I/O Fast Read (Winbond datasheet §7.2.16, p. 35) which is what the SDK's `boot2_w25q080.S` enables. The 256-byte budget is enforced by the Boot ROM: bytes 0–251 are the bootloader code, bytes 252–255 are the CRC32 (computed over bytes 0–251). The SDK's `pad_checksum` script computes and inserts the CRC after assembly. You do not write Stage 1 this week — we use the SDK's `boot_stage2_w25q080` library — but you read the 200 lines of ARM assembly on Tuesday and understand what each instruction does.
- **UF2 file format.** 512-byte blocks, 32-byte header (`magicStart0`, `magicStart1`, `flags`, `targetAddr`, `payloadSize`, `blockNo`, `numBlocks`, `familyID`-or-`fileSize`), 476 bytes of payload (256 used, 220 zero-padded for RP2040), 4-byte trailer (`magicEnd`). Cite Microsoft UF2 spec at <https://github.com/microsoft/uf2#file-format>. The RP2040 family ID `0xE48BFF56u` is registered in `utils/uf2families.json` in that repo. The Boot ROM's USB MSC mode accepts UF2 files exclusively; raw `.bin` files written to the RPI-RP2 drive are rejected silently. The Pico SDK's `elf2uf2` tool (in `tools/elf2uf2/`) handles the conversion; we will re-implement the parser side (Exercise 1) and the host-side packer (homework problem 4).
- **USB DFU 1.1 in summary.** Class code 0xFE, subclass 0x01, protocol 0x01 (run-time) or 0x02 (DFU mode). The DFU functional descriptor (Table 4.2, p. 14 of the spec) is 9 bytes: `bLength=9, bDescriptorType=0x21, bmAttributes (bitWillDetach + bitManifestationTolerant + bitCanUpload + bitCanDnload), wDetachTimeOut, wTransferSize, bcdDFUVersion=0x0110`. The seven requests on endpoint zero are tabulated in §3, p. 9. The state machine (Figure A.1, p. 38, plus textual rules in §6) has 10 states; for a download-only device, half the states (`dfuUPLOAD_IDLE`, etc.) are unreachable. Cite USB DFU 1.1 throughout the chapter; the spec is free.
- **The dual-bank layout.** Active bank vs staging bank; metadata page with a "swap requested" flag plus the staging image's version number, SHA256, and Ed25519 signature; the swap is atomic at the metadata-page level (the page either commits or it doesn't, because the device can lose power mid-write). The bootloader's startup checks the metadata first; if a valid swap is requested *and* the staging image's signature verifies *and* the SHA256 matches, the bootloader copies staging→active (4 KB at a time, page-aligned), clears the swap flag, and resets. Cite MCUboot's `BOOT_SWAP_TYPE_TEST` and `BOOT_SWAP_TYPE_PERM` discussion at <https://docs.mcuboot.com/design.html#swap-types>.
- **The linker script.** The Pico SDK's `memmap_default.ld` places `.text` at `0x10000100` (flash offset 256), `.data` at SRAM start (`0x20000000`), and `.bss` after `.data`. For our two-bank layout we need two linker scripts: `memmap_bootloader.ld` places the bootloader at `0x10000100` with a 32 KB limit; `memmap_app.ld` places the application at `0x10008100` (the active-bank start) with a 752 KB limit. The application's vector table at `0x10008100` is what the bootloader jumps to via `__ISB(); __set_MSP(*(uint32_t *)0x10008100); ((void(*)(void))*(uint32_t *)0x10008104)();` — reads SP from offset 0, reads PC from offset 4, sets the main stack pointer, then long-jumps to the reset handler.
- **The Boot ROM flash API.** A small function table at `*((uint16_t *)0x14)`, pointed to by a magic-number-validated header. The Pico SDK exposes these as `rom_func_lookup(rom_table_code('F','C'))` returning a function pointer cast. The functions we need: `_flash_range_erase(addr, count, block_size, cmd)`, `_flash_range_program(addr, data, count)`, `_connect_internal_flash`, `_flash_exit_xip`, `_flash_flush_cache`, `_flash_enter_cmd_xip`. Each is < 50 instructions in the ROM; the SDK wraps them with interrupt-masking and a guard against double-entry. Cite RP2040 datasheet §2.8.3.1 (p. 132) and §2.8.3.2 (p. 134).
- **Ed25519 verify.** The verify operation: given a public key `A` (32 bytes), a message `M` (the firmware bytes), and a signature `(R, S)` (32+32 bytes), compute `k = SHA512(R || A || M) mod L`, then check `8 * S * B == 8 * R + 8 * k * A` on the Edwards curve `edwards25519`. The "8 *" cofactor multiplication is RFC 8032's "cofactored" verification, equivalent to but slightly more permissive than the "cofactorless" version some libraries use. The signature is computed over the full firmware image (the active or staging bank's bytes, excluding the metadata page). We use `ed25519_sign_open` from ed25519-donna with the cofactored equation. ~12 KB of `.text` on Cortex-M0+ at `-Os`. Cite RFC 8032 §5.1 ("Ed25519ph, Ed25519ctx, and Ed25519", pp. 13–17) and the ed25519-donna repo.
- **The signed-image format.** A `.ccf` file is `[header: 32 bytes] [firmware bytes: N bytes] [signature: 64 bytes]`. The header: `magic=0x43434631u ('CCF1'), version=1, fw_size_le32, fw_sha256[32 bytes]`. Total file size is `32 + N + 64`. The signature covers the header (excluding itself) plus the firmware bytes. The bootloader's verifier reads the header, verifies SHA256, then verifies the Ed25519 signature against the project's public key. A real product would use a TLV-encoded header (MCUboot's approach) for extensibility; we use the fixed format to keep the parser under 100 lines.
- **The OTA receive protocol.** Plain-text line protocol over CDC; line-terminated by `\n`; binary payloads encoded as lowercase hex. Commands: `BEGIN <total-size-decimal> <sha256-hex>\n`, `CHUNK <offset-decimal> <len-decimal> <hex-bytes>\n`, `END\n`, `REBOOT\n`. Responses: `OK[ <ctx>]\n`, `ERR <code> <msg>\n`. Chunk size 256 bytes (the flash program-page size); 752 KB of firmware is ~3000 chunks. At ~5 KB/s (the slowest case, CDC + hex-encoding overhead + flash-write latency), upload takes ~150 seconds; with binary CDC chunks the upload time drops to ~30 seconds, but the protocol is harder to debug without a sniffer. We stick with hex for week 10.
- **Anti-bricking patterns.** The watchdog-confirmation pattern: the application sets a flag in metadata on first boot, after it has successfully initialized; the bootloader, on next reset, checks the flag — if set, the application proved it works, the bootloader leaves it alone; if not set, the bootloader rolls back to the previous-known-good image (which was archived in the staging bank during the original swap). Two-boot lifecycle: boot N is the test boot, boot N+1 is the confirmed boot, boot N+2 onwards is steady state. Cite the MCUboot doc's "Image swap with rollback" section.
- **picotool usage.** `picotool info -a` shows the program info and address; `picotool save app.bin` reads the entire flash into a `.bin`; `picotool load app.uf2` flashes a UF2; `picotool reboot -a` reboots into application mode, `-u` reboots into USB MSC mode, `-f` forces it even from a running program. We use `picotool save` to verify our bootloader's flash writes worked, and `picotool reboot -u -f` to recover when the bootloader misbehaves.

## What is intentionally out of scope this week

- **Secure boot with a root-of-trust in fuses.** Real production-grade secure boot uses one-time-programmable fuses (the RP2040 has 64 of these in the OTP — datasheet §2.16, pp. 250–261) to store a hash of the bootloader's public key, making the bootloader itself signed and unmodifiable. We do not touch OTP this week; our public key lives in flash as part of the bootloader's `.rodata`, so a sufficiently motivated attacker with physical access can re-flash the bootloader and own the device. The exercise of locking the bootloader's public key into OTP is left for Week 17 (`C7-week-17-security-and-attack-surface`).
- **Encrypted firmware.** Our `.ccf` format is signed but not encrypted — anyone who reads the flash via SWD can disassemble the firmware. AES-128-GCM encryption with a chip-unique key from OTP is the obvious next step; we skip it this week to keep the focus on the boot-and-swap mechanics.
- **A/B equal-size partitions with swap-and-revert.** We use the simpler "active + staging" layout where staging is overwritten on each update. A real product would use two equal-size partitions and a swap-and-revert algorithm (MCUboot's `BOOT_SWAP_TYPE_TEST`) so a failed first boot reverts automatically without losing the previous firmware. We implement watchdog-based rollback (a weaker but simpler form) and discuss MCUboot's full algorithm in Lecture 3.
- **Multi-process firmware.** Some bootloaders run a tiny TCP/IP stack to fetch firmware over Ethernet during boot. The RP2040 has no built-in Ethernet and no built-in WiFi (the Pico W's WiFi is a separate `cyw43` chip handled by the SDK's `pico_cyw43_arch` library — too heavy for a 32 KB bootloader). Our OTA path runs in the *application*, not the bootloader; the bootloader receives no network traffic. This is the standard split for resource-constrained devices.

## A note on bricking

You will brick the chip this week. Probably twice. Once will be a flash-write to the wrong address that wipes the second-stage bootloader; the recovery is `picotool reboot -u -f` followed by re-flashing a known-good UF2. The other will be an application whose `Reset_Handler` faults immediately, and the bootloader keeps jumping to it forever in a reset loop; the recovery is holding BOOTSEL during power-on, the Boot ROM drops to USB MSC mode, and you re-flash. The Pico is designed for this — the BOOTSEL recovery path is in mask ROM and cannot be removed by anything you do in flash. Embrace the bricking; it is the only honest way to learn what each layer of the boot sequence is responsible for.

The exception is the OTP fuses, which we are not touching this week. If you find yourself in Lecture 1 wondering "why don't we just lock the public key into OTP" — yes, that is what production does, and yes, it makes the bootloader truly tamper-evident, and yes, it also makes the chip permanently configured in a way that survives all your future mistakes. We will revisit OTP in Week 17 when we have a full security-engineering vocabulary; for now, the bootloader is mutable and that is by design — you will rewrite it 30 or 40 times in the next eight weeks of C7.
