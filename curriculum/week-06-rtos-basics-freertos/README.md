# Week 6 — RTOS Basics with FreeRTOS on the RP2040

> *An RTOS is not Linux made small. It is a scheduler, a handful of synchronization primitives, and a contract: every task you create has a worst-case execution time, a priority, and a stack you sized on purpose. The kernel does not guess. You tell it everything.*

Welcome to Week 6 of C7. Last week you re-implemented the Week 4 SPI/I²C driver stack in C++20 — `constexpr` peripheral addresses, concepts constraining the bus interface, CRTP for the device base class — and your `.elf` came in within 1 % of the C version on `.text` size. The Saleae captures were byte-identical to the C build. You proved, on disassembly, that `Gpio<18>::set()` is a single `str` instruction.

This week the discipline turns inside out. You stop writing the entire foreground loop yourself and you let a kernel do it. The kernel is FreeRTOS — about 9 000 lines of MIT-licensed C, the most widely-deployed real-time kernel on the planet, running on the Pi Pico W in 8 KB of flash and 4 KB of RAM. By Sunday you will have a three-task application running on top of it: one task reads a BMP280 over I²C on a 100 ms cadence, one task transforms the reading into a fixed-point Celsius value, one task prints the transformed value over UART at 115200 baud. The three tasks communicate through a queue and share the UART through a mutex. You will be able to demonstrate, on a logic analyzer, that the UART trace is uncorrupted even when the sensor task and the print task race for the bus.

An RTOS is the first piece of software in C7 that has a *worst-case timing contract* with you. Every API call in this lecture has a documented bound — `xQueueSend` is O(1), `xSemaphoreTake` with a timeout is bounded by the timeout plus a small constant, the context switch on Cortex-M0+ is ~84 cycles plus the cache effects (and Cortex-M0+ has no cache, so it is exactly 84 cycles). You will memorize a small set of those bounds. They are what makes "real-time" mean something more than "fast on average."

## Learning objectives

By the end of this week, you will be able to:

- **Define** the difference between a *hard* real-time system (deadline miss = system failure; e.g. anti-lock brakes, pacemaker pulse, motor commutation), a *firm* real-time system (deadline miss = output discarded but not catastrophic; e.g. a video frame in a decoder), and a *soft* real-time system (deadline miss = quality degraded; e.g. a display refresh). Place FreeRTOS-on-RP2040 firmly in the *hard* category for control loops and the *soft* category for housekeeping, and explain why the same kernel handles both.
- **Name** the three scheduling regimes a generic RTOS supports — *preemptive priority-based* (the default on FreeRTOS), *cooperative* (`configUSE_PREEMPTION = 0`), and *round-robin within a priority level* (`configUSE_TIME_SLICING = 1`) — and select the right one for a given workload. Cite the FreeRTOS Kernel documentation at <https://www.freertos.org/single-core-amp-smp-rtos-kernel.html> and the `FreeRTOSConfig.h` reference at <https://www.freertos.org/a00110.html>.
- **Create** a task with `xTaskCreate(vTaskCode, "name", usStackDepth, pvParameters, uxPriority, &pxTaskHandle)`, choose a stack depth justified by `uxTaskGetStackHighWaterMark` measurements, choose a priority justified by the rate-monotonic argument (faster cadence → higher priority), and explain what happens if you choose either one wrong (stack overflow corrupts the next task's TCB; priority inversion corrupts your timing budget). Cite the FreeRTOS API reference entry for `xTaskCreate`.
- **Read** the RP2040 datasheet §2.4 (Cortex-M0+, pp. 65–73) and locate the `SysTick` timer (pp. 96–98) that FreeRTOS uses as its tick source on this part. Explain that the tick is set in `FreeRTOSConfig.h` via `configTICK_RATE_HZ` (default on the Pico SDK port: 1 000 Hz, i.e. one tick per millisecond) and that *every* `vTaskDelay(pdMS_TO_TICKS(10))` is converted into ticks at compile time by that macro.
- **Use** a FreeRTOS *queue* (`xQueueCreate`, `xQueueSend`, `xQueueReceive`, `xQueueSendFromISR`) to pass a copy of a struct from an interrupt handler to a task. Explain why a queue is the right primitive for ISR-to-task hand-off (it is lock-free on the producer side when used from an ISR, it is bounded in size, and it provides natural backpressure when the consumer is slower than the producer). Cite the FreeRTOS API reference for each function.
- **Use** a FreeRTOS *binary semaphore* (`xSemaphoreCreateBinary`, `xSemaphoreGive`, `xSemaphoreTake`, `xSemaphoreGiveFromISR`) for ISR-to-task *signalling* (the ISR says "an event happened" and the task wakes; the task does not need to know how many events happened, only that ≥ 1 happened since it last checked). Distinguish from a counting semaphore (`xSemaphoreCreateCounting`), which preserves the count.
- **Use** a FreeRTOS *mutex* (`xSemaphoreCreateMutex`, `xSemaphoreTake`, `xSemaphoreGive`) — which is *not* the same as a binary semaphore — to protect a shared resource (the UART, an I²C bus, a `log_buffer[]` array). Explain the priority-inheritance protocol that the mutex implements automatically: when a lower-priority task holds the mutex and a higher-priority task tries to take it, the lower-priority task is *temporarily promoted* to the higher priority until it releases. Cite Sha, Rajkumar, and Lehoczky, "Priority Inheritance Protocols: An Approach to Real-Time Synchronization" (IEEE Trans. Computers, vol. 39 no. 9, Sep 1990).
- **Recognize** the *priority inversion* pathology by inspection — a high-priority task is blocked on a resource held by a low-priority task while a medium-priority task preempts the low-priority task and runs to completion, transitively starving the high-priority task. Demonstrate the pathology with three tasks on a Pi Pico W, capture the failure on a logic analyzer, then fix it by switching the binary-semaphore guard to a mutex and re-capture. This is Challenge 1.
- **Distinguish** the ISR-safe API surface (`*FromISR` suffix, no blocking) from the task-context API (no suffix, may block). Cite the FreeRTOS portmacro layer that defines `portYIELD_FROM_ISR(xHigherPriorityTaskWoken)` and explain why an ISR that wakes a higher-priority task must request a context switch on the way out — otherwise the wake takes effect on the next tick, not immediately. Cite the FreeRTOS Kernel documentation §"Interrupt management".
- **Detect** a stack overflow at runtime by enabling `configCHECK_FOR_STACK_OVERFLOW = 2` in `FreeRTOSConfig.h` (the "high water" check, which writes a pattern across the stack at task creation and verifies a few bytes near the limit on every context switch) and supplying `vApplicationStackOverflowHook(TaskHandle_t, char *)`. Demonstrate the trap on a deliberately-undersized stack and read the offending task's name out of the hook argument.
- **Size** a stack budget from measurement, not guess: run with a 1 024-word stack, log `uxTaskGetStackHighWaterMark(NULL)` periodically, observe the peak usage across all operating modes (idle, peak sensor read, error-path log dump), and re-size to peak + 100 words margin. Cite the FreeRTOS API reference for `uxTaskGetStackHighWaterMark`.
- **Build** the FreeRTOS-on-RP2040 kernel from the official FreeRTOS-Kernel repo at <https://github.com/FreeRTOS/FreeRTOS-Kernel>, against the `GCC/ARM_CM0` port, integrated into a CMake project that links against `pico_stdlib`. Provide the `FreeRTOSConfig.h` and `CMakeLists.txt` snippet (see the mini-project README).

## Topics covered

- **What an RTOS is and is not.** The half of the kernel's job is *scheduling* — picking which of N ready tasks runs at each tick. The other half is *synchronization* — providing queues, semaphores, and mutexes so the tasks can share data without race conditions. The kernel is *not* a general-purpose OS: it does not provide a filesystem (Pi Pico has no disk, by default), a network stack (you add lwIP if you want one), a shell, or virtual memory. It is a scheduler + IPC. That is the whole product.
- **Preemptive vs cooperative scheduling.** Preemptive: the kernel can take the CPU away from a running task on every tick (or every interrupt that makes a higher-priority task ready), and the running task does not have to consent. Cooperative: a task runs until it explicitly yields with `taskYIELD()` or blocks on an API. FreeRTOS on the RP2040 defaults to preemptive with time-slicing within a priority level. You can switch the entire kernel to cooperative by setting `configUSE_PREEMPTION = 0`, but you almost never want to — cooperative scheduling means a buggy task with an infinite loop hangs the entire system.
- **Priorities and the rate-monotonic argument.** Liu and Layland (1973) proved that for a set of N independent periodic tasks with deadlines equal to periods, the *rate-monotonic* assignment (faster cadence → higher priority) is optimal among fixed-priority schedulers — if any fixed-priority assignment can meet all deadlines, rate-monotonic will. We apply this as a rule of thumb in the mini-project: the 100 Hz sensor task gets priority 3, the 10 Hz transform task gets priority 2, the 1 Hz UART print task gets priority 1, and the idle task is implicit at priority 0.
- **Tasks, TCBs, and stacks.** A FreeRTOS task is a function `void f(void *param)` that runs in its own context, with its own stack, and its own copy of the Cortex-M0+ register file when it is *not* running. The kernel maintains a *Task Control Block* (TCB) per task — a struct containing the stack pointer, the task name, the priority, the state (Ready, Running, Blocked, Suspended), and a couple of list pointers for the scheduler's ready/blocked queues. The TCB is about 80 bytes on the M0+ port. The stack is separate — at minimum 128 words (512 bytes) per task on the M0+; we size for measured peak + margin.
- **The idle task and the idle hook.** The kernel creates a single low-priority idle task automatically. It runs whenever no other task is ready. By default it just spins. You can supply `vApplicationIdleHook()` and the kernel calls it every time it picks the idle task — common uses: tickling a watchdog, sleeping the CPU with `__WFI` to drop power consumption, or freeing dynamically-allocated task memory that was queued for cleanup. Set `configUSE_IDLE_HOOK = 1` to enable.
- **The tick.** The Cortex-M0+ has a built-in 24-bit `SysTick` countdown timer (RP2040 datasheet pp. 96–98). FreeRTOS programs it to fire at `configTICK_RATE_HZ` and uses the resulting ISR to decrement timeouts on blocked tasks, to handle time-slicing within a priority level, and to make decisions about which task to run next. The default rate on the Pico SDK port is 1 000 Hz, i.e. one tick per millisecond — high enough to give `vTaskDelay(pdMS_TO_TICKS(1))` a real meaning, low enough that the kernel's overhead is a few percent at most.
- **Queues.** A FreeRTOS queue is a FIFO of fixed-size elements, allocated at creation time, copied by value on send and receive. `xQueueSend(q, &item, timeout)` blocks the calling task until either space is available or the timeout expires. `xQueueReceive(q, &item, timeout)` is symmetric on the consume side. The `*FromISR` variants are non-blocking and accept a `BaseType_t *pxHigherPriorityTaskWoken` out-parameter, which the ISR must check on the way out and feed to `portYIELD_FROM_ISR`.
- **Semaphores: binary and counting.** A *binary* semaphore is a single-bit signal — `xSemaphoreGive` raises the bit, `xSemaphoreTake` lowers it (blocking until it is up). A *counting* semaphore is the same with a saturating count — `Give` increments, `Take` decrements (blocking when zero). Binary semaphores are for one-shot event notification ("an interrupt fired"). Counting semaphores are for resource pools ("there are N free buffers in this pool") or for event accumulation ("the ISR fired 5 times since the task last serviced it").
- **Mutexes and priority inheritance.** A mutex looks like a binary semaphore but enforces the rule "only the task that took it may give it back" — you cannot have task A take and task B give. More importantly, FreeRTOS mutexes implement *priority inheritance*: if a high-priority task blocks on a mutex held by a low-priority task, the low-priority task is temporarily promoted to the high priority until it releases the mutex. This bounds the worst-case priority inversion to the mutex hold time, which you can reason about. Binary semaphores have no priority-inheritance and *will* allow unbounded inversion if you (mis)use them for resource protection.
- **The ISR-safe API surface.** FreeRTOS draws a hard line between the task-context API and the ISR-context API. The ISR API has a `FromISR` suffix on every function, never blocks, and takes the `pxHigherPriorityTaskWoken` out-parameter. The task API never has the suffix and may block. Calling a task-context API from an ISR will trip a `configASSERT` (if assertions are enabled) or silently corrupt the scheduler's state (if they are not). Lecture 3 walks the boundary in detail.
- **Stack overflow detection.** The Cortex-M0+ has no MMU and no hardware stack-limit register. If a task's stack overflows, the next push wanders into another task's TCB or another task's stack — silent corruption, often surfacing minutes later as a wild branch. FreeRTOS provides two software-level checks: method 1 ("low water") compares the current stack pointer against the limit at every context switch; method 2 ("high water") additionally checks for a known pattern written across the stack at creation. Method 2 is more reliable and only slightly slower. Use method 2 (`configCHECK_FOR_STACK_OVERFLOW = 2`) in development; you may downgrade to method 1 in shipping firmware after the budgets are stable.
