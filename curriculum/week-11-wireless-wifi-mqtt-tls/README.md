# Week 11 — Wireless on the RP2040W: WiFi, MQTT, and TLS

> *Putting a radio on a microcontroller is the moment the embedded world stops being a closed system. Up to last week, every bug you fixed was reproducible on a single board with a single USB cable. This week, the board joins a network: it negotiates a four-way handshake with an access point it has never met before, opens a TCP connection through a router that may drop it at any time, derives session keys with a broker whose certificate chains back to a root you have never personally verified, and publishes a 96-byte JSON sensor reading every ten seconds. Every one of those steps can fail in a way you cannot see from the board's UART. The job this week is to make the firmware survive each of those failures cleanly — exponential backoff on a refused TCP connect, certificate-verification errors that abort the session instead of the firmware, MQTT keepalives that the broker actually accepts, and a publish loop whose ten-second cadence holds within ±200 ms over a week of operation.*

Welcome to Week 11 of C7 Crunch Wire. Last week your RP2040 ran a 32 KB bootloader that validated an Ed25519 signature over a 752 KB application image, copied it from staging to active across the dual-bank flash layout, and chain-loaded it on every cold reset — and the host-side `cc-flash` tool pushed firmware over CDC in under 30 seconds. You learned that the boot sequence is three stages (mask ROM at `0x00000000`, the 256-byte stage-2 bootloader the SDK ships, and your own application or bootloader at flash offset 256); that signed images make the difference between a firmware-update mechanism and an unauthenticated arbitrary-code-execution vector; and that anti-bricking is a *protocol* (watchdog confirmation, two-boot lifecycle, fallback partition) not a feature.

This week we add a radio. The board is the **Raspberry Pi Pico W** — the RP2040 paired with an Infineon CYW43439 single-chip WiFi 4 (802.11 b/g/n) plus Bluetooth 5.2 module on the same PCB. The CYW43 is a discrete chip with its own ARM Cortex-M3 inside, its own firmware blob, and an SDIO/SPI link to the RP2040; the RP2040 does *not* run the WiFi MAC/PHY itself. The `pico_cyw43_arch_lwip` library in the pico-sdk is the glue: it loads the CYW43 firmware over the bit-banged SPI driven from PIO state machine 1 on GPIO 24/25/29, exposes a small command interface (`cyw43_arch_wifi_connect_*`, `cyw43_arch_gpio_put` for the onboard LED which is wired to the CYW43 not the RP2040), and ties the CYW43's Ethernet-frame interface into the lwIP TCP/IP stack so your application code sees raw sockets. Cite pico-sdk's `pico_cyw43_arch` doxygen at <https://www.raspberrypi.com/documentation/pico-sdk/networking.html> and the CYW43 driver source at `lib/cyw43-driver/` in the SDK tree.

Above the CYW43 driver sits **lwIP** (lightweight IP, the embedded TCP/IP stack from Adam Dunkels' group at SICS, BSD-3-Clause, ~50k LOC). The pico-sdk vendors lwIP 2.2.0 in `lib/lwip/` and exposes its raw API (callback-driven, single-threaded, no `accept()` blocking — every operation is a registration that fires when bytes arrive). Above lwIP we layer **mbedTLS** (formerly PolarSSL, Apache-2.0, ~300k LOC) for TLS 1.2 client sessions; the pico-sdk vendors mbedTLS in `lib/mbedtls/` and provides a `pico_mbedtls` build target. And above mbedTLS we layer an MQTT 5 client; the SDK does *not* vendor an MQTT client, so we use Erich Styger's port of the Eclipse Paho embedded-C MQTT library (Eclipse Public License 2.0, ~3000 LOC) or — better — we write our own ~600-line MQTT 5 client because the protocol is small enough to do correctly in a week and that is exactly what we do in the mini-project.

The build target for the week is a **sensor node**. It joins WPA2/WPA3 WiFi on boot; opens a TLS 1.2 connection to the broker `test.mosquitto.org:8883`; CONNECTs as an MQTT 5 client with a randomized client-id; PUBLISHes a JSON payload (`{"temp_c":24.7,"vbat_v":3.28,"uptime_s":140}`) on the topic `cc7/devices/<id>/telemetry` every 10 seconds at QoS 0; handles SUBSCRIBE to `cc7/devices/<id>/cmd` for inbound commands (`reboot`, `set-interval <s>`); and recovers cleanly from every failure mode you can synthesize — AP power-cycle, broker disconnect, certificate-validation failure, TLS handshake timeout, DNS NXDOMAIN, TCP RST. The metric we hold: *median uptime publish-success ratio ≥ 99.0% over 24 hours, with a worst-case publish gap ≤ 60 s during a forced disconnect.* That number is the grade boundary.

The free public broker we publish to is **test.mosquitto.org**, Roger Light's public sandbox at <https://test.mosquitto.org/>. It accepts MQTT 3.1.1 and MQTT 5 on port 1883 (unencrypted) and 8883 (TLS); it is rate-limited at the broker level (no published SLA, but practically ~10 msg/s per client is fine); messages are *not* retained beyond ~30 seconds; and the broker's certificate chains back to ISRG Root X1 (Let's Encrypt). It is read-write but anyone can subscribe to anyone else's topics, so do not publish secrets. For development this is exactly what you need. A local Mosquitto install (`brew install mosquitto` / `apt install mosquitto`) on your laptop is the offline alternative and we cover the bring-up in resources.md.

The mini-project at the bottom of the week is a complete sensor node: ~1100 lines of C across 8 files (main, wifi, tls, mqtt-client, json, retry-backoff, sensor-mock, the lwIP callbacks header), plus the mbedTLS configuration header that pins the cipher list, the CA-bundle blob (the ISRG Root X1 PEM concatenated into a `const char[]`), and a 50-line Python `cc-listen.py` script that subscribes to your topic from a laptop and pretty-prints the readings. End to end, you plug the Pico W into a USB power supply (no laptop tether), it joins WiFi within 4 seconds of cold boot, publishes its first reading within 8 seconds, and runs for the week without intervention. The publish cadence holds at 10 s ± 200 ms; if WiFi drops, the firmware backs off (1 s, 2 s, 4 s, 8 s, 16 s, capped at 60 s) until reconnect; if the broker rejects the CONNECT (e.g. duplicate client-id), the firmware regenerates a new client-id and retries.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** the CYW43439 architecture and its link to the RP2040. The CYW43 is an Infineon (formerly Cypress, formerly Broadcom) single-chip 802.11 b/g/n WiFi 4 + Bluetooth 5.2 module; it contains a 160 MHz ARM Cortex-M3, ~512 KB of RAM, a 2.4 GHz RF front-end, and runs a closed-source firmware blob (~360 KB, distributed in the pico-sdk under `lib/cyw43-driver/firmware/` as a binary). The link to the RP2040 is a 4-wire SPI bus driven not by the SPI peripheral but by **PIO state machine 1**, because the CYW43's SPI variant is not standard SPI (it uses the same wire for data-in and data-out, half-duplex, with a gap-insertion timing requirement that PIO meets cleanly). The CYW43 driver in the pico-sdk (`cyw43_driver_pico_w` library) loads the firmware blob into the CYW43 over this PIO-SPI link on every boot — boot time is ~1.5 s, dominated by that load. Cite the pico-sdk repo's `lib/cyw43-driver/README.md` and the Pico W datasheet at <https://datasheets.raspberrypi.com/picow/pico-w-datasheet.pdf> (28 pages).
- **Use** the pico-sdk's `pico_cyw43_arch_lwip` API to join a WPA2 or WPA3 network. The minimal sequence: `cyw43_arch_init()` (loads firmware, ~1.5 s), `cyw43_arch_enable_sta_mode()`, `cyw43_arch_wifi_connect_timeout_ms(ssid, password, CYW43_AUTH_WPA2_AES_PSK, 30000U)`. On success the lwIP `netif` `cyw43_state.netif[CYW43_ITF_STA]` is up with a DHCP-assigned IPv4 address; on failure the call returns a non-zero error code (`PICO_ERROR_TIMEOUT`, `PICO_ERROR_CONNECT_FAILED`, `PICO_ERROR_BADAUTH`). The `pico_cyw43_arch_lwip_threadsafe_background` variant runs lwIP on a SYS_TICK background context; the `pico_cyw43_arch_lwip_poll` variant requires you to call `cyw43_arch_poll()` in your main loop. We use the `_poll` variant in the mini-project because it interacts more predictably with mbedTLS's blocking handshake. Cite the pico-sdk doc at <https://www.raspberrypi.com/documentation/pico-sdk/networking.html>.
- **Explain** the lwIP stack's raw API model. lwIP is a single-threaded callback-driven TCP/IP stack written for ~8 KB to ~16 KB of SRAM. Its three APIs (raw, netconn, socket) differ in threading model; the pico-sdk exposes the raw API only. Raw API: every operation is a non-blocking registration. Open a TCP connection with `tcp_new()`, call `tcp_connect(pcb, ip, port, connected_cb)`, return; when the SYN-ACK arrives, lwIP invokes `connected_cb(arg, pcb, err)` from its tick context. To send, `tcp_write(pcb, buf, len, TCP_WRITE_FLAG_COPY)` then `tcp_output(pcb)`. To receive, register `tcp_recv(pcb, recv_cb)`; when bytes arrive, `recv_cb(arg, pcb, pbuf, err)` fires; you process the `pbuf` chain and call `tcp_recved(pcb, len)` to advance the receive window. Memory management is the hard part: every `pbuf` must be `pbuf_free`-d exactly once. Cite lwIP 2.2.0 docs at <https://www.nongnu.org/lwip/2_2_x/raw_api.html>.
- **Walk** TLS 1.2 client-side from first byte to first encrypted application record. The TLS 1.2 handshake (RFC 5246 §7.3, <https://www.rfc-editor.org/rfc/rfc5246>) is six round-trips of records: ClientHello (cipher list + random + extensions like SNI for test.mosquitto.org), ServerHello + Certificate + ServerKeyExchange + ServerHelloDone, then ClientKeyExchange + ChangeCipherSpec + Finished, then server's ChangeCipherSpec + Finished. mbedTLS handles this end to end via `mbedtls_ssl_handshake()`, which we drive from `tls_handshake_step()` calls in our retry-backoff loop. Cite mbedTLS docs at <https://mbed-tls.readthedocs.io/en/latest/kb/how-to/mbedtls-tutorial/> and RFC 5246 directly. TLS 1.3 (RFC 8446) collapses this to one round-trip and is a future-week topic; pico-sdk's vendored mbedTLS supports TLS 1.3 but the Pico's CPU at 125 MHz makes the RSA-PSS verification on the server cert noticeably slower than ECDSA — we stick with TLS 1.2 + ECDHE-RSA-AES128-GCM-SHA256 which handshakes in ~1.2 s on cold connect.
- **Configure** mbedTLS for an embedded MQTT client. The default mbedTLS build includes hundreds of cipher suites, all named curves, both TLS 1.2 and 1.3, OCSP, ALPN, key-share, and 30+ certificate-parsing paths — and weighs ~250 KB of flash. The pico-sdk's `pico_mbedtls` target builds against your project's `mbedtls_config.h`, in which you `#undef` everything you do not need. Our config: one cipher suite (`MBEDTLS_TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256`), one curve (`MBEDTLS_ECP_DP_SECP256R1_ENABLED`), TLS 1.2 only (`MBEDTLS_SSL_PROTO_TLS1_2`, undef the rest), no client certificates (`#undef MBEDTLS_X509_CSR_PARSE_C`), no OCSP, no ALPN. Result: mbedTLS in our build is ~95 KB of flash (compared to ~250 KB stock). We pin the broker's CA (ISRG Root X1) as a trust anchor — no system CA bundle on an MCU; we ship the one CA we need. Cite the mbedTLS configuration guide at <https://mbed-tls.readthedocs.io/en/latest/kb/how-to/how-do-i-configure-mbedtls/>.
- **Implement** an MQTT 5 client from the wire spec. MQTT 5 is OASIS Standard "MQTT Version 5.0" (free at <https://docs.oasis-open.org/mqtt/mqtt/v5.0/mqtt-v5.0.html>, 167 pages, dated 2019-03-07). The wire format: every packet starts with a 1-byte fixed header (4-bit packet type + 4-bit flags), followed by a Variable Byte Integer (1-4 bytes) encoding the remaining length, followed by the variable header, followed by the payload. The 15 packet types are CONNECT (`0x10`), CONNACK (`0x20`), PUBLISH (`0x30`), PUBACK (`0x40`), PUBREC, PUBREL, PUBCOMP, SUBSCRIBE (`0x82`), SUBACK (`0x90`), UNSUBSCRIBE, UNSUBACK, PINGREQ (`0xC0`), PINGRESP, DISCONNECT (`0xE0`), AUTH (new in MQTT 5). MQTT 5 adds Properties (a tagged key-value list inside the variable header) which encodes things like Session Expiry Interval, Receive Maximum, Topic Alias Maximum, User Property. For a publish-only sensor node we use four packet types: CONNECT, PUBLISH (QoS 0), PINGREQ, DISCONNECT. The full client we build also handles SUBSCRIBE+SUBACK and inbound PUBLISH for command channels. Cite the spec sections we lean on: §3.1 CONNECT (pp. 38–53), §3.3 PUBLISH (pp. 64–78), §3.12 PINGREQ (pp. 122–123), §3.14 DISCONNECT (pp. 125–131).
- **Compare** MQTT 5 against MQTT 3.1.1 and MQTT-SN. MQTT 3.1.1 is OASIS 2014 (<https://docs.oasis-open.org/mqtt/mqtt/v3.1.1/mqtt-v3.1.1.html>), shipped in millions of devices, no Properties, no reason codes in CONNACK (just a one-byte return code), maximum payload 256 MB but in practice no broker handles > 1 MB cleanly. MQTT 5 adds Properties, Reason Codes on all ACK packets, Topic Aliases (publish "1" instead of "very/long/topic/name" after registering the alias), Shared Subscriptions (`$share/group/topic`), and Will Delay. MQTT-SN ("MQTT for Sensor Networks", <https://www.oasis-open.org/committees/document.php?document_id=66091&wg_abbrev=mqtt>, 2013) is a UDP-friendly fork for low-power radio networks (Zigbee, LoRa) where TCP is unaffordable; payloads are encoded with topic IDs instead of strings to fit single 802.15.4 frames. We use MQTT 5 because the Pico W has TCP available and MQTT-SN is for a constraint we do not have. Cite the comparison in the OASIS FAQ.
- **Design** a retry-backoff state machine for embedded networking. Four failure types: (1) WiFi-association failure (AP rejects the join — wrong password, MAC filtering, AP overloaded); (2) DHCP failure (no IPv4 lease within 10 s); (3) DNS NXDOMAIN or TCP-connect refusal (broker hostname does not resolve, or broker refuses the SYN); (4) TLS handshake failure (cert chain does not validate, handshake times out); (5) MQTT CONNECT rejected (client-id collision, broker overloaded). Backoff strategy: exponential with jitter — first retry at 1 s, then 2 s, 4 s, 8 s, 16 s, cap at 60 s, with ±25% uniform jitter on each interval (so a swarm of devices rebooting after a power outage does not retry-synchronize and DoS the broker). Cite the "Exponential Backoff and Jitter" article from AWS Architecture Blog at <https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/>. The state machine has five states (`STATE_BOOT`, `STATE_WIFI_DOWN`, `STATE_WIFI_UP_BROKER_DOWN`, `STATE_CONNECTED`, `STATE_BACKOFF`) and you draw it on the whiteboard before writing the code.
- **Reason** about the cost of CA-chain storage on an MCU. A single Let's Encrypt root certificate (ISRG Root X1) in PEM form is ~2 KB of text; in DER form (parsed by mbedTLS at boot) ~1.4 KB; in the parsed in-memory `mbedtls_x509_crt` structure ~3 KB of RAM (because each ASN.1 OID is heap-allocated). Multiply by a typical desktop's ~150 trusted roots and you have ~450 KB of RAM, which is more than the RP2040 has. The answer on an MCU is **trust anchor pinning**: ship the one CA your one broker uses. If the broker's CA rotates, you reflash the firmware. This is operationally painful and is what TLS 1.3's `certificate_authorities` extension is meant to fix, but at MCU scale the only realistic option is "small number of pinned CAs that you control the rotation of." For test.mosquitto.org the chain is `Server cert → R3 (Let's Encrypt intermediate) → ISRG Root X1`; pinning the root suffices because mbedTLS will walk the intermediate from the server's certificate-chain extension. Cite the Mosquitto certificate page at <https://test.mosquitto.org/ssl/> for the current CA.
- **Diagnose** wireless and TLS failures from the inside out. Diagnostics in order of usefulness: (1) `cyw43_arch_wifi_connect_timeout_ms` return code — tells you whether WiFi associated; (2) the lwIP `netif`'s `ip4_addr_isany(&netif->ip_addr)` check — tells you whether DHCP completed; (3) `dns_gethostbyname` callback `err` — tells you whether DNS resolved; (4) `tcp_connect` callback `err` — tells you whether the SYN got a SYN-ACK; (5) `mbedtls_ssl_handshake` return code — tells you whether the TLS handshake completed; (6) MQTT CONNACK `Reason Code` — tells you whether the broker accepted the connection. Each of these is a checkpoint; logging them with a timestamp lets you read off the failure point from the UART without a sniffer. For deeper diagnosis: Wireshark on a separate laptop running as a WiFi AP (macOS's Internet Sharing or `hostapd` on Linux) sniffs every packet your Pico sends — and TLS handshakes are partly visible (you see the ServerHello and the cipher chosen even though the application data is encrypted).
- **Measure** publish cadence and connection-loss recovery on the bench. The grade-boundary metric is *median uptime publish-success ratio ≥ 99.0% over 24 hours, with a worst-case publish gap ≤ 60 s during a forced disconnect.* You measure this by logging every PUBLISH with its `(timestamp, sequence_number)` on the device side (printed over UART) and matching against the `cc-listen.py` subscriber's received-timestamp log on the laptop side; the ratio is `received / sent`. Force-disconnect tests: power-cycle the AP for 5 s, 30 s, 5 min; observe backoff intervals and reconnect time. The 99% bar leaves headroom for ~14 missed publishes per 24 hours (8640 expected publishes per day at 10 s cadence); a single 60 s AP outage costs 6 publishes; you have budget for two of those per day. Cite the metric design choice as ours.
- **Build** a complete sensor node and a host-side subscriber tool. The mini-project is 8 source files (~1100 LOC of C), one mbedTLS config header, one CA-bundle header (the ISRG Root X1 PEM as a string literal), one CMakeLists.txt, and the 50-line `cc-listen.py` host-side subscriber. End to end on the bench: cold boot to first publish in ≤ 8 s, sustained 10 s cadence, ≥ 99% delivery over 24 h, ≤ 60 s gap during forced AP outage. Budget: total firmware size ≤ 500 KB at `-Os` (which fits comfortably in the 752 KB active bank from Week 10's bootloader; you can deploy this week's mini-project through last week's bootloader and it Just Works).

## Topics covered

- **The CYW43439 hardware.** Infineon single-chip 802.11 b/g/n WiFi 4 + Bluetooth 5.2; ARM Cortex-M3 inside running closed-source firmware (~360 KB blob shipped in the pico-sdk); link to the RP2040 is a 4-wire SPI bus driven from PIO state machine 1 because the timing requirements are non-standard. The onboard LED on the Pico W is wired to the CYW43, not the RP2040 — `gpio_put(25, 1)` does nothing on a Pico W; you must `cyw43_arch_gpio_put(CYW43_WL_GPIO_LED_PIN, 1)` instead. Power consumption: ~80 mA average while associated and idle, ~250 mA peak during transmit. Cite Pico W datasheet at <https://datasheets.raspberrypi.com/picow/pico-w-datasheet.pdf> (28 pages).
- **The pico-sdk's `pico_cyw43_arch_lwip_poll` build target.** Pulls in the cyw43 driver, lwIP 2.2.0, and a small "arch" shim that adapts lwIP's `sys_*` and `pbuf_*` primitives to the RP2040. Your main loop calls `cyw43_arch_poll()` which services the CYW43 link, then `cyw43_arch_wait_for_work_until(absolute_time)` to sleep until the next event or the next timer expiry. The threading model is single-threaded cooperative; you never see lwIP callbacks fire from interrupt context. Cite the SDK doxygen.
- **WiFi association.** `cyw43_arch_wifi_connect_timeout_ms(ssid, password, auth_mode, timeout_ms)`. Auth modes: `CYW43_AUTH_OPEN`, `CYW43_AUTH_WPA_TKIP_PSK`, `CYW43_AUTH_WPA2_AES_PSK`, `CYW43_AUTH_WPA2_MIXED_PSK`, `CYW43_AUTH_WPA3_SAE_AES_PSK`, `CYW43_AUTH_WPA3_WPA2_AES_PSK`. WPA3-SAE is supported by the CYW43 firmware version shipped in pico-sdk 1.5.1 and later (early Pico W boards had WPA2-only firmware; the driver auto-updates the blob on every boot). The four-way handshake completes in ~800 ms on a typical home AP; DHCP DISCOVER → OFFER → REQUEST → ACK takes another ~500 ms; total time from `cyw43_arch_init()` to `ip4_addr_isany() == false` is ~3 s.
- **lwIP's raw TCP API.** `tcp_new()`, `tcp_bind()`, `tcp_connect(pcb, ip, port, connected_cb)`, `tcp_write(pcb, buf, len, TCP_WRITE_FLAG_COPY)`, `tcp_output(pcb)`, `tcp_recv(pcb, recv_cb)`, `tcp_recved(pcb, len)`, `tcp_close(pcb)`. Callbacks: `err_t connected_cb(void *arg, struct tcp_pcb *pcb, err_t err)`, `err_t recv_cb(void *arg, struct tcp_pcb *pcb, struct pbuf *p, err_t err)` (p == NULL means peer FIN-d), `err_t sent_cb(void *arg, struct tcp_pcb *pcb, u16_t len)`, `void err_cb(void *arg, err_t err)` (fatal error, pcb is already freed — *do not call tcp_close from err_cb*). Memory: every `pbuf` received in recv_cb must be `pbuf_free`-d after the data is consumed. Cite lwIP 2.2.0 raw API docs at <https://www.nongnu.org/lwip/2_2_x/raw_api.html>.
- **DNS resolution.** `dns_gethostbyname(name, addr_out, found_cb, arg)`. If the name is already in the cache, returns `ERR_OK` and `*addr_out` is filled in; if a lookup must be issued, returns `ERR_INPROGRESS` and the callback fires when the response arrives. We resolve `test.mosquitto.org` once at startup and cache the IPv4 result for the session; if the TCP connect fails, we re-resolve in case the broker has rolled to a new IP.
- **TLS 1.2 handshake walkthrough.** ClientHello (SNI = `test.mosquitto.org`, supported_versions = [TLS 1.2], cipher_suites = [ECDHE-RSA-AES128-GCM-SHA256], elliptic_curves = [secp256r1], signature_algorithms = [RSA-PKCS1-SHA256]) → ServerHello → Certificate (server cert + intermediate R3) → ServerKeyExchange (ephemeral ECDH public key, signed by the server cert's RSA private key) → ServerHelloDone → ClientKeyExchange (client's ephemeral ECDH public key) → ChangeCipherSpec → Finished → ChangeCipherSpec → Finished. After Finished, both sides have derived the same `master_secret` from `pre_master_secret = ECDHE_shared_secret`, plus `client_random` and `server_random`. Cite RFC 5246 §7.3 (pp. 33–38) and §8 (pp. 47–50).
- **mbedTLS configuration trimming.** A `mbedtls_config.h` for embedded MQTT clients: define `MBEDTLS_SSL_CLI_C`, `MBEDTLS_SSL_PROTO_TLS1_2`, `MBEDTLS_PKCS1_V15`, `MBEDTLS_AES_C`, `MBEDTLS_GCM_C`, `MBEDTLS_ECDH_C`, `MBEDTLS_ECDSA_C`, `MBEDTLS_SHA256_C`, `MBEDTLS_BIGNUM_C`, `MBEDTLS_X509_CRT_PARSE_C`, `MBEDTLS_RSA_C`, `MBEDTLS_ECP_C`, `MBEDTLS_ECP_DP_SECP256R1_ENABLED`, `MBEDTLS_KEY_EXCHANGE_ECDHE_RSA_ENABLED`. Undef: `MBEDTLS_SSL_SRV_C`, `MBEDTLS_SSL_PROTO_TLS1_3`, `MBEDTLS_SSL_PROTO_TLS1`, `MBEDTLS_SSL_PROTO_SSL3`, `MBEDTLS_X509_CSR_PARSE_C`, OCSP, ALPN, every cipher other than AES-GCM, every curve other than secp256r1. Result: mbedTLS in our build is ~95 KB of `.text` (vs ~250 KB stock).
- **The MQTT 5 wire format.** 1-byte fixed header (high nibble = packet type, low nibble = flags), Variable Byte Integer (VBI) for remaining length (1–4 bytes, 7 bits of data per byte with the high bit as a continuation flag), variable header (per-packet), payload (per-packet). CONNECT variable header: protocol name = "MQTT" (4-byte UTF-8 string), protocol level = `0x05`, connect flags (1 byte: clean-start, will, will-qos, will-retain, password, username, reserved-0), keep-alive (2 bytes), Properties (VBI length + tagged key-value list). CONNECT payload: client-id (UTF-8 string), then will-properties + will-topic + will-payload if will-flag is set, then username + password if those flags are set. We hand-encode this in `mqtt_encode.c` (~250 LOC).
- **MQTT 5 Properties.** Each property is a 1-byte identifier from MQTT 5 Table 2-4 (pp. 23–25 of the spec) followed by a value whose format depends on the identifier (Byte, 2-byte Integer, 4-byte Integer, VBI, Binary Data, UTF-8 String, UTF-8 String Pair). The properties we use: `Session Expiry Interval` (`0x11`, 4-byte integer) in CONNECT to control session lifetime; `Receive Maximum` (`0x21`, 2-byte integer) in CONNECT to limit in-flight QoS > 0 messages; `Topic Alias Maximum` (`0x22`, 2-byte integer) in CONNECT to advertise alias support; `Reason String` (`0x1F`, UTF-8 string) in DISCONNECT to log why we are leaving. Cite §2.2.2 Properties (pp. 22–28).
- **The retry-backoff state machine.** `STATE_BOOT` → `STATE_WIFI_DOWN` (cyw43_arch_init + connect_timeout_ms). On success → `STATE_WIFI_UP_BROKER_DOWN` (DNS + tcp_connect + mbedtls_ssl_handshake + MQTT CONNECT). On success → `STATE_CONNECTED` (the publish loop). On any failure → `STATE_BACKOFF`, which sleeps `backoff_ms` with jitter then retries from `STATE_WIFI_DOWN` if WiFi is the failure, or from `STATE_WIFI_UP_BROKER_DOWN` if WiFi was fine. `backoff_ms` doubles on each consecutive failure, capped at 60000U; resets to 1000U on each successful CONNECT. Cite AWS exponential-backoff-with-jitter article.
- **The JSON encoder.** A 60-line `json_emit.c` that produces compact JSON without a heap (writes into a caller-provided buffer; returns the byte count). Supports objects, arrays, strings, integers, fixed-point decimals (no `printf("%f")` because the float-printf code is ~12 KB of additional `.text`; we format `int_part.fraction_part` manually). For our 96-byte sensor reading this is plenty.
- **The CA-bundle pin.** ISRG Root X1, the Let's Encrypt root that signs test.mosquitto.org's intermediate. We embed the PEM as a 2-KB string literal in `ca_bundle.h`, `mbedtls_x509_crt_parse`-it once at boot, and pass it to `mbedtls_ssl_conf_ca_chain`. We also document the rotation procedure: every June 2030, ISRG Root X1 expires; you reflash with ISRG Root X2 or the next root before that date. Cite the Let's Encrypt chain-of-trust page at <https://letsencrypt.org/certificates/>.
- **Bench measurement.** Log every PUBLISH on the device side as `[+12347 ms] PUB seq=1234 bytes=96`; on the laptop side `cc-listen.py` logs `[+12.349 s] RECV seq=1234 bytes=96`. Reconcile the two logs at the end of a 24-h run; the delivery ratio is `len(received_seqs ∩ sent_seqs) / len(sent_seqs)`. Force-disconnect tests: AP power-cycle for 5 s, 30 s, 5 min; expected gaps in the received log are within backoff envelope (≤ 60 s).
