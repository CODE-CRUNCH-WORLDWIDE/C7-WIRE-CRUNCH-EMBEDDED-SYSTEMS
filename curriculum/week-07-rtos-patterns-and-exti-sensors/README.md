# Week 7 — RTOS Patterns and EXTI-Driven Sensor Reads on the RP2040

> *Polling is a confession that you do not know when the data will arrive. An interrupt is a contract with the silicon: when this pin moves, this code runs, before anything else. The RTOS lets you keep the contract by deferring the work — the ISR posts a token, a task wakes, the task does the slow part. Three lines of code, six pages of consequences.*

Welcome to Week 7 of C7. Last week you stood up FreeRTOS on the RP2040, ran three tasks at three priorities, and proved on a Saleae capture that the UART trace stays clean even when the sensor task and the print task race for the bus. Your `STACK-BUDGET.md` showed each task within +100 words of its measured peak, your `FAULT-MODEL.md` documented the four kernel-specific failure modes you defended against, and you could state — without looking it up — why a mutex is not the same thing as a binary semaphore.

This week the polling loop comes out. The BMP280 in the Week 6 mini-project did not have a useful `INT` pin, but the next sensor on the bench does: an MPU-6050 IMU on `i2c1`, with an `INT` line wired to `GP22`. Every time the MPU has a fresh sample (configurable, 1 Hz to 1 kHz; we will run at 200 Hz), it pulls `INT` high for 50 µs. That edge is going to fire a hardware interrupt on the RP2040, the ISR is going to call `xSemaphoreGiveFromISR`, and a task blocked on `xSemaphoreTake(portMAX_DELAY)` is going to wake up, drain the sensor's FIFO over I²C, and post the result to a stream buffer. The transform task at the other end of the stream buffer is going to consume a variable-length stream of bytes — not fixed-size structs — and emit a 100 Hz attitude estimate. The system has no polling loops anywhere. The CPU spends idle time in `__WFI`. Power draw drops by ~30 %.

The deeper material this week is the *pattern catalogue*. Last week you knew three primitives: task, queue, mutex. This week you learn six more: **stream buffer**, **message buffer**, **direct-to-task notification**, **gatekeeper task**, **deferred-interrupt task**, and **bounded-mailbox / overwrite-on-full**. You learn which to pick when. You learn that `xQueueSendFromISR` of a 64-byte struct is about 6× more expensive than `xStreamBufferSendFromISR` of the same 64 bytes, because the queue does a `memcpy` and the stream buffer does a `memmove` into a ring. You learn that direct-to-task notifications are 45 % faster than a binary semaphore on the same hand-off because they bypass the queue/semaphore layer entirely. These trade-offs are documented in the FreeRTOS reference manual §3 and §5; we will cite them as we go and you will reproduce two of the benchmarks on bench.

You will also learn the *real* priority-inversion bound. Last week you saw the pathology and the fix on a binary-semaphore vs mutex swap. This week you derive the bound formally: with N tasks and M mutexes, the worst-case priority-inversion latency a task of priority *p* can experience is bounded by the sum over all lower-priority tasks of (longest critical section they hold on any mutex *p* might want). Sha-Rajkumar-Lehoczky (1990) proved this is tight; we re-derive on paper and verify on the bench by constructing a worst-case scenario in Exercise 3.

The mini-project assembles every Week 7 primitive into one application: an MPU-6050 IMU samples at 200 Hz via EXTI on `GP22`, an attitude-estimation task consumes a stream buffer and emits a 100 Hz quaternion, a gatekeeper task owns the UART and formats every log line, and the entire system runs for an hour with sub-millisecond worst-case ISR-to-task latency measured on a logic analyzer.

## Learning objectives

By the end of this week, you will be able to:

- **Distinguish** the four FreeRTOS data-passing primitives by their underlying data structure and their cost: a *queue* (`xQueueCreate`) is a fixed-element-size FIFO that copies by value, ~1.4 µs send on Cortex-M0+ at 125 MHz for 12-byte items; a *stream buffer* (`xStreamBufferCreate`) is a byte-oriented ring with no element boundaries, ~0.6 µs send for the same payload; a *message buffer* (`xMessageBufferCreate`) is a stream buffer that prepends a length header so consumers receive whole messages; a *direct-to-task notification* (`xTaskNotifyGive`, `ulTaskNotifyTake`) is a 32-bit value (or counter, or bit-mask) attached to the task's TCB, no allocation, ~0.4 µs to signal. Cite the FreeRTOS reference manual §4 (Queue Management), §5 (Stream and Message Buffers), §11 (Task Notifications) and reproduce the four numbers on your own bench.
- **Wire** an external sensor interrupt into the RP2040 NVIC. The MPU-6050 `INT` pin goes to `GP22`; you configure `GP22` with `gpio_set_irq_enabled_with_callback(22, GPIO_IRQ_EDGE_RISE, true, &gpio_callback)`; the SDK installs the shared GPIO ISR; your callback runs in IRQ context with NVIC priority 0xC0 (configurable). Cite RP2040 datasheet §2.4.5 (NVIC) pp. 99–104, §2.19.6 (GPIO IRQ) pp. 281–284, and the pico-sdk `hardware_gpio` reference at <https://www.raspberrypi.com/documentation/pico-sdk/group__hardware__gpio.html>.
- **Reason** about NVIC priority numbering on Cortex-M0+: only 2 bits of priority, four levels (0, 0x40, 0x80, 0xC0), lower number = higher priority. FreeRTOS critical sections mask interrupts at or above `configMAX_SYSCALL_INTERRUPT_PRIORITY` (default 0xC0 on the Pico SDK port — i.e. priority 3, the lowest); interrupts at higher hardware priority (0x00 through 0x80) remain enabled but *cannot* call any FreeRTOS API. Document why this is the right default and what it costs. Cite the FreeRTOS portmacro layer (`portable/GCC/ARM_CM0/portmacro.h`) and the ARMv6-M ARM §B3.4 (System Control Space).
- **Implement** the deferred-interrupt-handling pattern: ISR is three lines (read source, give semaphore / send to stream buffer, yield); the actual sensor read happens in a task at priority *p* − 1 where *p* is the ISR's effective priority. Quantify the worst-case ISR-to-task latency: ISR entry + ISR body + `portYIELD_FROM_ISR` context-switch path ≈ 4.2 µs on Cortex-M0+ at 125 MHz, dominated by the `PendSV` context-switch (84 cycles per FreeRTOS porting guide, plus ~120 cycles of save/restore on top of the M0+ register file). Measure on bench, compare to the budget. Cite "Mastering the FreeRTOS Real Time Kernel" Chapter 6 ("Interrupt Management").
- **Use** `xStreamBufferSendFromISR` to push a variable-length byte stream from an ISR to a task. Explain the trigger-level mechanism (`xStreamBufferSetTriggerLevel`): the consumer task does not wake until at least N bytes are available, which avoids waking the consumer 12 times to deliver 12 single-byte chunks. Tune the trigger level for the MPU-6050 case where each interrupt produces 14 bytes (1 status byte + 6 accel bytes + 1 temp byte + 6 gyro bytes) and the consumer wants whole frames. Cite the FreeRTOS reference manual §5.3.
- **Distinguish** when to use a *binary semaphore* (event signalling, lose-count-on-multiple-fire — accept that the consumer may miss intermediate events) from a *counting semaphore* (resource pool or event accumulation — the count survives multiple ISRs between consumer wakeups) from a *direct-to-task notification* in counting mode (same as counting semaphore but ~3× faster and no allocation). The MPU-6050 fires at 200 Hz; if the consumer task takes a 6 ms hiccup it will miss one event with a binary semaphore but not with a counting primitive. Test this empirically in Exercise 2.
- **Apply** the *gatekeeper task* pattern: a single task owns a resource (the UART, an SPI bus, a logging ring), and every other task posts to its queue. No mutex needed — the only task that touches the resource is the gatekeeper. Compare with the mutex pattern from Week 6: the gatekeeper has slightly higher latency (a queue send + a context switch) but removes the priority-inversion concern entirely and removes the possibility of a non-gatekeeper task forgetting to unlock the mutex. Cite "Mastering the FreeRTOS Real Time Kernel" §7.7.
- **Derive** the priority-inversion bound under priority-inheritance: for task *T* of priority *p*, with mutexes M1..Mk that lower-priority tasks may hold, the worst-case blocking time *B* on *T* is bounded by the sum of the longest critical section any single lower-priority task holds on any mutex *T* might request, summed over the mutexes — formally, *B*(*T*) ≤ Σᵢ max{ csᵢⱼ : lⱼ < p, lⱼ may take Mᵢ, p may also take Mᵢ }. Re-derive on paper in Lecture 2 and verify on bench in Exercise 3. Cite Sha, Rajkumar, Lehoczky (IEEE TC 1990, pp. 1175–1185).
- **Recognize** the *deadlock-by-mutex-order* pathology — two tasks each take two mutexes in opposite order — and apply the textbook prevention rule (acquire mutexes in a globally-fixed order, by address or by ID). Demonstrate the failure in Challenge 1, document it, fix it.
- **Configure** `__WFI` in the FreeRTOS idle hook to drop CPU power consumption from ~85 mA to ~58 mA at 125 MHz. Explain that `WFI` halts the core until the next interrupt (any interrupt, including the SysTick), measure power on the Pico's 3V3 line with a multimeter or INA219, and document the measurement in `POWER.md`. Cite RP2040 datasheet §2.4.7 (power management) and ARMv6-M ARM §A6.7.74 (`WFI`).
- **Build** an end-to-end IMU pipeline: MPU-6050 interrupt → ISR → stream buffer → estimator task → gatekeeper task → UART. The total budget from interrupt edge to formatted line ready in the UART TX FIFO is 2 ms at the 99-th percentile, measured by GPIO toggle at each stage and read on a Saleae 8.

## Topics covered

- **The deferred-interrupt-handling pattern.** Restate the core idea: an ISR has a hard contract to be short. Cortex-M0+ has no nested-interrupt support in the simple sense (it does support tail-chaining but not preemption within the same priority), so the time spent inside an ISR is time *all other* ISRs at the same or lower priority wait. The rule is therefore: ISR does the minimum work necessary to acknowledge the source and signal a worker task; the worker task does the slow part. FreeRTOS provides four primitives for the signalling step (binary semaphore, counting semaphore, direct-to-task notification, stream/message buffer); we will compare them.
- **NVIC priority on Cortex-M0+.** Two implementation bits of priority means four levels: 0x00 (highest), 0x40, 0x80, 0xC0 (lowest). The RP2040's NVIC is documented in datasheet §2.4.5 (pp. 99–104). FreeRTOS sets `configMAX_SYSCALL_INTERRUPT_PRIORITY` to 0xC0 by default on the Pico port; this means *every* interrupt the SDK installs (UART, I²C, GPIO, DMA, etc.) is at priority 0xC0 (FreeRTOS-API-callable) and the kernel reserves 0x00–0x80 for *truly* deadline-critical work that does not need any kernel call (a tight commutation loop, a sigma-delta DAC update). We will not put any ISR above 0xC0 this week. We will discuss why a project might.
- **GPIO interrupts on the RP2040.** Every GPIO can fire interrupts on `RISE`, `FALL`, `LEVEL_HIGH`, `LEVEL_LOW`. All 30 GPIOs share two NVIC slots (`IO_IRQ_BANK0` and `IO_IRQ_QSPI`); the SDK installs a shared dispatcher that reads `IO_BANK0.PROC0_INTS0..3` to find the source. Datasheet §2.19.6 (pp. 281–284). Latency from edge to ISR entry: ~24 cycles of NVIC handshake plus ~16 cycles of dispatch, ≈ 320 ns at 125 MHz. We measure this in Exercise 1.
- **Stream buffers.** A FreeRTOS stream buffer is a single-producer / single-consumer ring of bytes (the SPSC restriction is real — multiple producers or multiple consumers require external synchronization). `xStreamBufferSend` copies bytes in; `xStreamBufferReceive` copies bytes out. The trigger level controls when a blocked consumer wakes: consumer wakes when `bytes_available >= trigger_level`. The default trigger level is 1 byte (wake on every send), which is rarely what you want. Stream buffers are byte-oriented; if you want whole-message semantics, use a *message buffer* — same data structure with a 4-byte length prepended per record. Cite the FreeRTOS reference manual §5.
- **Direct-to-task notifications.** The lightest-weight signal mechanism. Every task has a 32-bit notification slot in its TCB; `xTaskNotifyGive` increments it (or sets bits, or overwrites it, depending on the action argument), `ulTaskNotifyTake` decrements (or reads and clears, or reads). No queue allocation, no separate object, ~0.4 µs per signal on M0+. Use when the signaller knows exactly which task to wake. The trade-off: a task can only have one notification slot, so multi-event signalling needs the bit-mask mode or a separate primitive. Cite the reference manual §11.
- **Gatekeeper tasks.** The textbook resource-protection pattern that does not use a mutex. A single task owns the resource; every other task posts requests to its queue. Advantages: no priority-inheritance complexity (the owner runs at one fixed priority); no deadlock-on-mutex-order possibility; the resource access is naturally serialized by the queue's FIFO semantics. Disadvantage: one extra context switch per access. We use a UART gatekeeper in the mini-project and compare against the Week 6 mutex-protected UART.
- **The priority-inheritance bound.** Sha-Rajkumar-Lehoczky proved that with the priority-inheritance protocol (which FreeRTOS mutexes implement), the worst-case blocking time on a high-priority task is the sum of one critical section per mutex (the longest one held by any lower-priority task on that mutex). This is a known, bounded quantity — you can compute it by static inspection of the source. Without priority inheritance, the bound is unbounded (any number of medium-priority tasks can intercede). Without inheritance the system is unanalyzable.
- **Deadlock by mutex order.** Two tasks, two mutexes, opposite order: task A holds M1 and waits for M2; task B holds M2 and waits for M1. Both wait forever. The classical fix: impose a total order on mutex acquisition (e.g. always take the lower-address mutex first), enforced by code review or by a static-analysis tool. We demonstrate the bug in Challenge 1 and the fix.
- **`__WFI` in the idle hook.** Cortex-M0+ supports `WFI` (Wait For Interrupt), which halts the CPU clock until any interrupt fires. Plugging `__wfi();` into `vApplicationIdleHook()` drops the Pico's 3V3 current from ~85 mA to ~58 mA at 125 MHz — a 30 % power saving for typing two characters of code. The trade-off: the time to wake from `WFI` adds 1–3 cycles to the worst-case ISR latency. For a 200 Hz sample rate this is invisible; for a 100 kHz control loop it is not. Cite RP2040 datasheet §2.4.7 and ARMv6-M ARM §A6.7.74.
- **EXTI vs polling: the latency math.** A polling loop at 1 kHz has worst-case 1 ms latency on a 1 kHz-sample sensor (you may have just missed an edge). An EXTI line has worst-case ~5 µs latency on the same sensor (NVIC handshake + ISR body + context switch). The 200× improvement is paid for in code complexity: you must handle the ISR, you must size the stream buffer, you must reason about NVIC priority. The trade-off is worth it for any sensor faster than ~10 Hz; below that, polling is simpler.
